\chapter{Bevezetés}

\section{A téma aktualitása és célja}

Kis és középvállalatoknál végzett szakmai munkáim során gyakran azt tapasztaltam, hogy a cégnél használt IT eszközök (például számítógépek, laptopok, mobiltelefonok stb.) nem megfelelő nyilvántartása megnehezíti a mindennapi feladatok elvégzését az IT osztályon belül és esetlegesen anyagi károk is jelenthet a munkaadó számára.

A digitalizáció a mai munkaerőpiacon alapkövetelménynek tekintető, ennek következtében az intézményekben használt korábban említett eszközök darabszáma is drasztikusan megnőtt. Az igények folyamatos változása és a használatban lévő eszközök darabszámának növekedése miatt ezeknek az IT eszközöknek a nyilvántartása létfontosságú a vállalati életben.

Tapasztalataim alapján jellemző a különböző táblázatkezelő alkalmazások használata (például Microsoft Excel), melyeknek hátránya, hogy az adott felhasználó minden adatot szabadon tud szerkeszteni, a meglévő adatstruktúrát módosíthatja. Egy közepes méretű cég esetén ez akár több száz különböző eszközt is jelenthet, amiknek a mozgását, státuszát és felelősét nyilvántartani egy táblázatkezelőben nem fenntartható hosszú távon. Az ilyen megoldások alkalmazásakor gyakori az adatvesztés, idővel a struktúra teljes összeomlása és az "Egyéb" mezők rendszeres megjelenése az "adatbázisban".

Szakdolgozatom célja, egy olyan webalkalmazás fejlesztése ami a korábban említett problémákra biztosít egy lehetséges megoldást.
A program egy olyan környezetet szolgáltat a felhasználóknak és az IT munkatársaknak amin keresztül a felhasználók egyszerűen tudnak új eszközöket igényelni, megtekinthetik a jelenlegi eszközeiket és állapotukat, a vezetők jóváhagyási folyamaton keresztül tudják felülvizsgálni az alkalmazottak igényeit, az adminisztrátorok pedig teljes körű rálátást kapnak a telephelyen lévő összes eszközre.

Az alkalmazásom által az eszközök teljes életciklusa digitálisan nyomon követhetővé válik, ezáltal csökken az adminisztrációs teher és minimalizálható az emberi hibákból eredő adatvesztés.

A tervezés során az alábbi célokat tűztem ki:

\begin{itemize}
	\item Eszközök (számítógépek, telefonok, SIM kártyát) kezelése
	\item Igénylési folyamat megvalósítása
	\item Jóváhagyási folyamat megvalósítása
	\item Igénylés feldolgozásának megvalósítása
	\item Átadási / Átvételi dokumentum digitalizációja
	\item Használatban lévő eszközök életciklusának követhetősége
	\item Felhasználó és jogosultság kezelés
\end{itemize}

Célom, hogy az alkalmazásom bevezetésével növelem az átláthatóságot és hosszú távon csökkenthetem az üzemeltetésre, adminisztrációra fordítót kapacitás mennyiségét.
A fejlesztés során személyes célom, hogy elmélyítsem a tudásom a piacképes technológiák használatában és az alkalmazásfejlesztés vállalati szintű gyakorlatában.

\subsection{Microsoft .NET 8 Web API}

A szerveroldali logika megvalósításához a .NET 8 Web API keretrendszert alkalmaztam.

A .NET egy ingyenes, platformfüggetlen, nyílt forráskódú fejlesztői platform, amely különféle alkalmazások építésére nyújt lehetőséget. 
A nagy teljesítményű futtatókörnyezetre támaszkodik, amelyet számos nagy léptékű alkalmazás használ éles környezetben.
A .NET platform a termelékenység, a teljesítmény, a biztonság és a megbízhatóság biztosításához készült. \cite{DotnetBevezetés}

Az alkalmazásban az Entity Framework Core (EF Core) ORM (Object-Relational Mapping) keretrendszer használatával valósítottam meg az adatbázis műveletek egyszerű és biztonságos kezelését.

Az Entity Framework (EF) Core a népszerű Entity Framework adatelérési technológia egyszerűsített, bővíthető, nyílt forráskódú és platformfüggetlen verziója.

Az EF Core objektum-relációs leképezőként (O/RM) szolgálhat, amely:

\begin{enumerate}
	\item Lehetővé teszi, hogy a .NET-fejlesztők .NET-objektumokkal dolgozzanak egy adatbázissal.
	\item Kiküszöböli az általában írandó adatelérési kódok többségének szükségességét.
\end{enumerate}

Az EF Core használatával az adathozzáférés egy modell használatával történik. A modell entitásosztályokból és környezeti objektumból áll, amelyek az adatbázissal való munkamenetet jelölik. A környezeti objektum lehetővé teszi az adatok lekérdezését és mentését.
\cite{EntityFrameworkCore}

\chapter{A rendszer célja és funkcionális követelményei}

Az alkalmazásom célja, hogy a vállalati környezetben használt informatikai eszközöknek (jelenleg számítógépek, laptopok, telefonok, SIM kártyák) biztosítson egy átlátható és könnyen karbantartható felületet.

A megvalósítás során fontos szempont volt, hogy az alkalmazás egyszerűen használható, a jövőbeli igényeknek megfelelően bővíthető legyen.

Elvárt követelmény, hogy az alkalmazás bevezetése után a felhasználók adminisztrációs feladatait csökkenti, kapacitást szabadít fel.

\section{A rendszer célja}

A rendszer alapvető célja, hogy \textbf{digitalizálja és automatizálja az informatikai eszközök nyilvántartását}, valamint megkönnyítse a felhasználók munkáját.
Ennek érdekében a fejlesztés során az alábbi célokat határoztam meg:

\begin{enumerate}
	\item Egységes eszköznyilvántartás létrehozása
	
	Az összes informatikai eszköz adatai egy központi adatbázisban kerülnek tárolásra.
	Az eszközről tárolt információ, jelenlegi állapota és felelőse valós időben követhető.
	
	\item Eszközrendelési és jóváhagyási folyamat kialakítása 
	
	A dolgozók az alkalmazáson keresztül adhatják le új eszköz igényüket, amelyet a kiválasztott vezető bírál el.
	A rendelések feldolgozását az adminisztrátorok végzik.
	A rendelés feldolgozásához szükséges információkat és eszközöket egy felületen érnek el.
	Ez a digitális folyamat helyettesíti a korábbi e-mailes vagy szóbeli engedélyezéseket.
	
	\item Karbantartható és biztonságos rendszer fejlesztése
	
	A jogosultságkezelés több szintű: a rendszer megkülönbözteti a \textit{felhasználó}, a \textit{vezető} és az \textit{adminisztrátor} szerepköröket.
	
	Minden felhasználó csak a saját jogosultsági körébe tartozó funkciókat érheti el.
\end{enumerate}

\section{Funkcionális követelmények}

A tervezés során az alábbi funkcionális követelmények kerültek meghatározásra, szerepkörök szerint csoportosítva.

\subsection{Bejelentkezés és hitelesítés}

A rendszer használata kizárólag hitelesített felhasználók számára engedélyezett. Ennek megfelelően az alkalmazás minden funkciója csak bejelentkezést követően érhető el.

A felhasználók a bejelentkezési felületen e-mail címük és jelszavuk megadásával azonosítják magukat. A megadott adatok ellenőrzését a szerver végzi.

Sikeres hitelesítés esetén a rendszer hozzáférést biztosít a felhasználó számára a jogosultsági szintjének megfelelő funkciókhoz. Sikertelen bejelentkezés esetén a rendszer hibaüzenetet jelenít meg, és nem engedélyezi a további műveletek végrehajtását.

A bejelentkezési folyamat biztosítja, hogy az alkalmazás funkciói kizárólag jogosult felhasználók számára legyenek elérhetők.

\subsection{Felhasználó}

\begin{itemize}
	\item Eszközigénylés létrehozása
	
	A felhasználók a rendelkezésükre álló felületeken keresztül új igényléseket adhatnak le. A számítógépek, laptopok és mobilkészülékek igényléséhez két különálló felület került kialakításra.
	
	Az igénylés rögzítéséhez az alábbi mezők kitöltése szükséges:
	
	\begin{itemize}
		\item Eszközkategória:
			
		A felhasználó legördülő listából választhatja ki a kívánt eszközkategóriát, amelyeket az adminisztrátorok hoznak létre és tartanak karban. A kiválasztott kategória meghatározza az eszköz típusát.
		
		\item Ügyféltípus:
		
		Mobil eszköz igénylése esetén választható mező, amely az adminisztrátorok által definiált értékeket tartalmazza. Az egyes ügyféltípusok eltérő előfizetési csomagokat jelentenek.
		
		\item Átvétel helye:
		
		Az eszköz átvételének helyszíne, amely az elérhető telephelyek közül választható ki.
		
		\item Megjegyzés:
		
		Opcionálisan kitölthető mező, amely lehetőséget biztosít további információk megadására.
		
		\item Jóváhagyó:
		
		A vezetői jogosultsággal rendelkező felhasználók listáját tartalmazó választható mező.
		
	\end{itemize}
	
	Az igénylés rögzítését követően a felhasználó visszajelzést kap a felületen, továbbá értesítés kerül kiküldésre az igénylő és a kijelölt jóváhagyó részére.
	
	\item Igénylések nyomon követése
	
	A felhasználók a számukra elérhető felületen ellenőrizhetik a korábban leadott igénylések adatait, valamint azok aktuális státuszát.
	
	A különböző eszköztípusokhoz tartozó igénylések elkülönített felületeken jelennek meg.
	
	A főoldalon a leadott igénylések táblázatos formában, az alábbi adatokkal kerülnek megjelenítésre:
	
	\begin{itemize}
		\item Igénylés azonosító
		
		Minden igénylés egyedi azonosítóval rendelkezik.
		
		\item Eszköz típusa
		
		\item Átvétel helye
		
		\item Státusz
		
		\item Részletek megtekintése gomb
		
		Az adott igénylés részletes adatainak megjelenítése.
		
		\item Törlés gomb
		
		Amennyiben a feldolgozás még nem kezdődött meg, az igénylés törölhető.
	\end{itemize}
	
	\item Értesítések megtekintése
	
	A felhasználók a navigációs sáv megfelelő menüpontján keresztül érhetik el értesítéseiket. Az olvasatlan értesítések száma a felületen külön jelölésre kerül.
	
	Az értesítések listában jelennek meg, ahol az új elemek kiemelten láthatók.
	
	Értesítés az alábbi eseményekről történik:
	
	\begin{enumerate}
		\item Igénylés sikeres rögzítése
		\item Igénylés jóváhagyásra vár
		\item Igénylés elbírálása (jóváhagyás vagy elutasítás)
		\item Feldolgozás megkezdése
		\item Feldolgozás befejezése, eszköz átvehető
		\item Eszköz átadása
	\end{enumerate}
	
	\item Saját eszközök megjelenítése 
	
	A felhasználók számára elérhető egy eszköztípusonként strukturált felület, ahol saját eszközeik nyilvántartását tekinthetik meg.
	
	A megjelenítés táblázatos formában történik, az alábbi adatokkal:
	
	\begin{enumerate}
		\item Egyedi azonosító
		\item Kategória
		\item Modell
		\item Sorozatszám
		\item Státusz
		\item Státusz indoklása
	\end{enumerate}
	
	A felületen lehetőség van cserefolyamat kezdeményezésére, illetve az eszköz leadásának indítására.
	
\end{itemize}

\subsection{Vezető}

A vezetői szerepkörrel rendelkező felhasználók minden felhasználói funkcióval rendelkeznek, továbbá jogosultak a jóváhagyási folyamat kezelésére.

A jóváhagyási funkciók a navigációs sáv megfelelő menüpontján keresztül érhetők el, amely kizárólag ezen szerepkör számára látható.

A vezető kizárólag azokat az igényléseket tekintheti meg, amelyek esetében jóváhagyóként került kijelölésre.

A lista táblázatos formában jelenik meg, és tartalmazza a szükséges adatokat, valamint a részletes nézet elérését biztosító lehetőséget.

A részletes nézetben az igénylés jóváhagyható vagy elutasítható.
A döntésről az igénylő értesítést kap.
Jóváhagyás esetén az igénylés az adminisztrációs folyamatba kerül.

\subsection{Adminisztrátor}

Az adminisztrátori jogosultsággal rendelkező felhasználók a navigációs sáv menüpontjain keresztül érhetik el funkcióikat, amelyek kizárólag számukra elérhetők.

\begin{itemize}
	\item Felhasználók kezelése
	
	Felhasználói fiókok létrehozása és karbantartása.
	
	\item Eszköznyilvántartás kezelése
	
	A készleten lévő eszközök rögzítése, valamint adataik és státuszuk naprakészen tartása.
	
	\item Eszközkategóriák kezelése
	
	Az igénylés során választható kategóriák létrehozása, módosítása és törlése.
	
	\item Igénylések feldolgozása
	
	Az adminisztrátor feladata a jóváhagyott igénylések feldolgozása.
	
	A feldolgozás során az adminisztrátor kiválaszt egy megfelelő, készleten lévő eszközt, majd azt hozzárendeli az adott igényléshez. Mobil eszköz esetén a készülék mellett SIM-kártya hozzárendelése is szükséges.
	
	A hozzárendelést követően az igénylés státusza „Átvehető” értékre módosul, és a felhasználó értesítést kap.
	
	Az eszköz átvételét követően az adminisztrátor lezárja az igénylést, amelynek státusza „Kész” értékre változik. Ettől a ponttól kezdve az eszközért a felhasználó felel.
	
	
\end{itemize}

	\item \textbf{Teljesítmény:} A .NET 8 és a React technológiák révén az alkalmazás gyors válaszidőt biztosít még nagyobb adatbázis esetén is.
	\item \textbf{Reszponzivitás:} A frontend mobilbarát kialakítású, így bármilyen eszközről kényelmesen használható.
	\item \textbf{Karbantarthatóság:} A komponensalapú React frontend és a rétegzett .NET architektúra biztosítja a könnyű bővíthetőséget.
	\item \textbf{Megbízhatóság:} A fejlesztés során az API végpontokat tesztelésekkel és hibakezeléssel is megerősítettem.
\end{itemize}

\section{Rendszerarchitektúra}
A fejlesztett alkalmazás háromrétegű architektúrát követ, amely jól elkülöníti a megjelenítési, az üzleti logikai és az adatkezelési rétegeket

A modell legnagyobb előnye, hogy lehetővé teszi az egyes rétegek egymástól függetlenül történő fejlesztését, sőt, akár teljes cseréjét is, lépést tartva így a folyamatosan változó követelményekkel és az egyre újabb technológiákkal.
Ez biztonságosan megtehető, mert egy réteg módosítása nincs hatással a többi réteg működésére. Egymástól független modulokként tartalmazza a felhasználói felületet, az üzleti logikát és az adatbázist a szükséges hozzáférési műveletekkel. \cite{HáromrétegűArchitektúra}

A rendszer fő komponensei a következők:

\begin{itemize}
	\item Frontend (kliensoldal): A felhasználói felület React keretrendszer használatával készült. A frontend feladata az interakciók kezelése, az adatok megjelenítése, valamint az API hívások indítása a backend felé.
	
	\item Backend (szerveroldal): A szerveroldali logikát .NET 8 Web API biztosítja. Ez a réteg kezeli az üzleti logikát, feldolgozza a kliens kéréseit, valamint kommunikál az adatbázissal.
	
	\item Adatbázis: Az adatok tárolása Microsoft SQL Server adatbázisban történik, amely strukturált és megbízható adattárolást biztosít.

\section{JSON Web Token működése}

A JSON Web Token (JWT) egy nyílt szabvány (RFC 7519), amely kompakt, önmagát tartalmazó módot biztosít a felek közötti biztonságos információcserére JSON objektumként. Gyakran használják modern, állapot nélküli (stateless) webes alkalmazásokban hitelesítésre (authentication) és engedélyezésre (authorization), mivel a token magában hordozza a felhasználói adatokat, aláírása révén pedig hamisíthatatlan. \cite{JWT} 

A token felépítése:
\begin{itemize}
	\item Header
	
	A header rész általában két elemet tartalmaz, a token típusát és az alkalmazott aláírási algoritmust például HMAC SHA256 vagy RSA.
		
	\item Payload
	
	A token második része a payload, amely az úgynevezett állításokat (claims) tartalmazza. Az állítások egy adott entitásra -- jellemzően a felhasználóra -- vonatkozó információkat, valamint további adatokat foglalnak magukba. Az állítások három fő típusa különböztethető meg: regisztrált (registered), nyilvános (public) és privát (private) állítások.
	
	\item Signature
	
	Az aláírás létrehozásához a titkosított fejlécet, a titkosított payloadot, egy titkos kulcsot, valamint a fejlécben megadott algoritmust kell felhasználni, majd ezek alapján történik az aláírás generálása.
	
\end{itemize}

\section{Fejlesztési környezet}
A fejlesztés során modern és széles körben használt eszközöket alkalmaztam a hatékony munkavégzés érdekében. A backend fejlesztése Microsoft Visual Studio 2022 környezetben történt, amely teljes körű támogatást nyújt a .NET alapú alkalmazások fejlesztéséhez. A frontend fejlesztés Visual Studio Code szerkesztőben zajlott, amely könnyű, gyors és kiválóan alkalmas JavaScript alapú projektek kezelésére.

A verziókezeléshez Git rendszert használtam, amely lehetővé tette a fejlesztési folyamat követését, valamint a kód biztonságos tárolását. A forráskód GitHub platformon került tárolásra.

Az alkalmazás futtatásához és teszteléséhez Postman eszközt is használtam az API végpontok ellenőrzésére.

\section{Backend fejlesztés}
A backend megvalósítása .NET 8 Web API keretrendszer segítségével történt. Az alkalmazás REST alapú szolgáltatásokat biztosít, amelyek HTTP kéréseken keresztül érhetők el.

A rendszer több különálló vezérlőt (controller) tartalmaz, amelyek az egyes erőforrások kezeléséért felelősek. Ilyenek például:

\begin{itemize}
	\item DeviceController – eszközök kezelése
	\item OrderController – rendelések kezelése
	\item AuthController – felhasználói hitelesítés
\end{itemize}

A backend működésének alapja a rétegzett architektúra, amelyben a controller réteg a kérések fogadásáért, míg a repository réteg az adatbázis műveletekért felel.

Egy tipikus működési folyamat a következő:

\begin{enumerate}
	\item A kliens HTTP kérést küld egy adott végpontra.
	\item A controller fogadja a kérést és továbbítja a megfelelő szolgáltatásnak.
	\item A repository réteg lekérdezi vagy módosítja az adatokat az adatbázisban.
	\item A backend JSON formátumban visszaküldi a választ a kliensnek.
\end{enumerate}

Az adatkezeléshez Entity Framework Core keretrendszert használtam, amely lehetővé teszi az objektum-orientált adatbázis-kezelést. A lekérdezések LINQ segítségével valósulnak meg.

A biztonság érdekében a felhasználók hitelesítése JWT tokenekkel történik, és az API végpontok védettek jogosultságellenőrzéssel.

\section{Frontend fejlesztés}
A frontend fejlesztése React könyvtár segítségével történt. Az alkalmazás egy egyoldalas alkalmazásként (Single Page Application, SPA) működik, amely dinamikusan tölti be az adatokat a backend API-ból.

A felület komponensalapú felépítésű, amely lehetővé teszi az egyes elemek újrafelhasználását és egyszerű karbantartását. A főbb komponensek közé tartoznak:

\begin{itemize}
	\item bejelentkezési felület
	\item eszközlista megjelenítő komponens
	\item rendelési űrlap
	\item adminisztrációs felület
\end{itemize}

A navigációt React Router biztosítja, amely lehetővé teszi az oldalak közötti gyors váltást újratöltés nélkül.

Az API hívások Axios könyvtáron keresztül történnek. A frontend a backend által biztosított REST végpontokat használja az adatok lekérésére és módosítására.

Az állapotkezelés a React hook-jain keresztül valósul meg, például useState és useEffect használatával. Ezek segítségével a felhasználói interakciók és az adatok változásai hatékonyan kezelhetők.

A felhasználói felület kialakításakor fontos szempont volt az egyszerű kezelhetőség és az átlátható megjelenés, amelyet Tailwind CSS segítségével valósítottam meg.

\section{Integráció}
A frontend és a backend közötti integráció REST API-n keresztül valósul meg. A kliensoldali alkalmazás HTTP kéréseket küld a backend végpontjai felé, amelyek feldolgozzák a kéréseket és visszaküldik a szükséges adatokat.

Az autentikáció során a frontend eltárolja a bejelentkezéskor kapott JWT tokent, majd minden további API hívás során elküldi azt az Authorization fejlécben.

A rendszer biztosítja a hibakezelést is: amennyiben egy API hívás sikertelen, a frontend megfelelő visszajelzést ad a felhasználónak.

Az integráció során külön figyelmet fordítottam arra, hogy a frontend és backend közötti adatcsere egységes és jól strukturált legyen.

\section{Biztonság és naplózás}
A rendszer biztonságos működése érdekében több védelmi mechanizmust alkalmaztam.

A felhasználók hitelesítése JWT token alapú megoldással történik, amely biztosítja, hogy csak jogosult felhasználók férhessenek hozzá az API végpontokhoz.

A jelszavak nem kerülnek tárolásra nyílt formában, hanem titkosított (hash-elt) módon kerülnek az adatbázisba. Ez jelentősen növeli az adatok biztonságát.

A rendszer tartalmaz alapvető hibakezelést is, amely biztosítja, hogy a váratlan hibák ne okozzanak rendszerleállást.

A naplózás (logging) lehetővé teszi a rendszer működésének nyomon követését, valamint a hibák későbbi elemzését. A naplózott események közé tartozhatnak például a bejelentkezések, hibák és fontosabb műveletek.
